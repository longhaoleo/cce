# 阶段性实验总结 Prompt

适用场景：

- 一轮实验做完了，想整理阶段总结
- 不是旧实验归档，而是刚做完一个阶段，想沉淀方法、结果和下一步
- 想把代码、输出、日志和思路收成 README / log / retrospective / handoff 文档
- 想复用一种“先看主线、再还原过程、再讲逻辑、最后给结论与下一步”的整理方式

可直接复制下面这段给 Codex / ChatGPT 使用。

```text
请把这次任务当成“阶段性实验总结”来处理。

这个实验阶段可能是：
- 正在推进中的一个活跃方向
- 刚结束、需要阶段复盘的一轮实验
- 暂停中的支线
- 准备归档的旧尝试

不要预设它一定是旧实验，也不要预设它一定要并入当前主线。先根据仓库证据判断它现在处于什么状态。

目标：
1. 先看仓库当前主线和上下文，判断这个实验阶段与主线的关系。
2. 阅读目标目录及相关代码、配置、输出、日志、必要时的 git 痕迹，反推出这一个阶段到底做了什么。
3. 把这个阶段整理成一份可复用的总结文档，默认落成 README、阶段总结、实验日志或 handoff 文档中最合适的一种。
4. 总结时不要只写结果，还要讲清楚方法逻辑，包括：
   - 这个阶段的目标 / 假设
   - 入口脚本、主要流程和关键改动
   - 数据/张量/中间结果是怎么流动的
   - 关键函数、关键步骤、关键指标分别在做什么
   - 输出目录和主要文件怎么解读
   - 这一阶段哪些尝试有效、哪些无效、哪些结论还不稳
   - 这一阶段对当前主线的意义
   - 下一步最合理的延续方式
5. 保留真实过程和历史信息，避免把过程压扁成“好像一开始就知道答案”的总结。

执行要求：
- 先看仓库根目录 README、相关 log/README、目标目录 README，再深入代码和输出。
- 解释必须以实际文件、代码路径、产物和日志为依据，不要脑补不存在的结论。
- 如果目录里有 notebook checkpoint、实验脚本、配置、生成产物、csv、图像、日志，都纳入分析。
- 说明时要区分“明确事实”和“根据现有证据做出的推断”。
- 不要只说这个阶段“做了什么”，还要说“为什么这么做”“这样做的局限是什么”。
- 默认优先补文档，不随意移动或删除大体量实验产物，除非我明确要求。
- 写作风格偏“给未来的自己/队友留档”，重点是方便后续接着做，而不是包装成汇报稿。

请按这个顺序工作：
1. 判断这个阶段和当前主线的关系
2. 总结这个阶段的目标、范围和状态
3. 还原这阶段的真实执行逻辑
4. 提炼有效发现、失败点、局限和待验证点
5. 把文档补完整，并明确下一步建议

如果你认为合适，请直接动手修改文件，不用只给方案。
```

## 推荐附加参数

如果你偏向“阶段复盘”，可以在末尾追加：

```text
请把文档写成“阶段目标 -> 实施逻辑 -> 结果与判断 -> 下一步”的结构。
```

如果你偏向“交接继续做”，可以在末尾追加：

```text
请额外写清楚：现在已经确认了什么、还有哪些不确定点、下一位继续做时应先看哪里。
```

如果你偏向“归档说明”，可以在末尾追加：

```text
如果你判断这条线不再是当前主线，请明确写出它为什么更适合被保留为归档或支线。
```
